Review the pull request.

## Instructions

1. Read `CONSTITUTION.md` for the non-negotiable security, concurrency, and data-integrity rules.
2. Read the root `CLAUDE.md` plus the per-directory `CLAUDE.md` for the area(s) the diff touches (`lotsa/`, `rigg/`) for conventions and architecture.
3. Read `.claude/skills/review/checklist.md` for the full review checklist with categories, gates, and suppressions.
4. Get the full diff: `git fetch origin main --quiet && git diff origin/main`
5. For each changed file, read the full file to understand context — do not review from the diff alone.
6. Apply the checklist against the diff: Pass 0 (intent), Pass 1 (critical categories), then Pass 2 (informational).

## Confidence filter

For every potential issue, silently rate your confidence it is a real bug (not a style nit,
not already handled, not a false positive). Only report issues at 7/10 confidence or above.

## DO NOT report these

- **Hypothetical future concerns** — "if X ever changes, Y would break." Review the code as it IS, not as it might become. If the concern requires a code change that hasn't happened, it's not a bug.
- **Input validation for trusted internal values** — UUIDs, task IDs generated by the system, config values from known sources. Don't flag missing validation on values the system itself generates.
- **"Worth noting" observations** that don't need fixing — if you wouldn't reject the PR over it, don't report it.
- **Style preferences** that Ruff handles — import ordering, blank lines, naming conventions.
- **Timing/sleep fragility in tests** — note only if a test is actually flaky, not hypothetically.

## Output format

Output your review as your final text response in markdown format.
Do NOT use gh pr comment — the action posts your output automatically.

### PR Review

**Verdict:** (choose ONE based on these rules)
- **CHANGES REQUESTED** — if any High-severity issue exists
- **ISSUES FOUND** — if only Medium or Low issues exist
- **APPROVED** — if no issues found

**Severity mapping** (from checklist gate classification):
- **High**: Pass 1 categories — Injection & Data Safety, Credential & Secret Safety, Cryptographic Correctness, Audit Trail Integrity
- **Medium**: Pass 2 categories — Concurrency & Atomic Transitions, Performance & Async Hygiene, API/CLI Discipline, Database Conventions, Frontend Patterns, SDK Boundary (lotsa↔rigg), Test Gaps, Dead Code, Type Safety
- **Low**: concrete issues that are low risk but real (not hypothetical "what if" concerns)

#### High
- [`file:line`] Description. Fix: suggestion.

#### Medium
- [`file:line`] Description. Fix: suggestion.

#### Low
- [`file:line`] Description (informational only).

Omit empty severity sections. Be terse — one line per issue.
Do not report style issues — Ruff handles those.
Respect the suppressions list in the checklist.
If no issues found, say: "No issues found."

End with a **Confirmed good** section listing security and correctness
patterns you verified and found correct (e.g., "CAS via atomic_transition with
result.won checked before side effects", "No credential exposure in error paths").
